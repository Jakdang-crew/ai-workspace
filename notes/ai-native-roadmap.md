## 1. 병목 진단 프레임워크

워크플로우 실패의 근본 원인 4가지 분류:

### 원인 A: 프롬프트 설계 문제

- **증상**: 같은 프롬프트인데 실행할 때마다 결과가 다름
- **디버깅**: temperature 0으로 5회 실행, 출력 분산 측정
- **해결**: XML 태그로 구조화, Few-shot 예시 3개(정상/엣지/실패) 포함

### 원인 B: 컨텍스트 윈도우 / 메모리 관리 문제

- **증상**: 워크플로우 후반에 앞선 지시사항을 잊음
- **디버깅**: 각 단계에서 "현재 목표와 제약조건을 요약해봐" 질문
- **해결**: 핵심 지시사항 상단/하단 양쪽 배치, 컨텍스트 컴팩션 패턴

### 원인 C: 에이전트 오케스트레이션 구조 문제

- **증상**: 개별 태스크 성공, 연결 시 실패
- **디버깅**: 각 단계 입력/출력 로그로 데이터 변질 지점 추적
- **해결**: JSON Schema로 태스크 간 인터페이스 명시, 단일 책임 에이전트 파이프라인

### 원인 D: 툴 호출 신뢰성 문제

- **증상**: 도구 호출 누락 또는 잘못된 파라미터
- **디버깅**: 도구 호출 로그에서 누락/파라미터 오류/결과 오해석 분류
- **해결**: 사용 조건·파라미터·반환값 명시, 자동 재시도 + fallback 설계

## 2. 단계별 스킬 업그레이드

### 1단계 (1~2개월): 워크플로우 안정화

**목표**: 10번 실행하면 10번 동일한 품질의 결과

- 프롬프트 엔지니어링 심화 (Role → Context → Task → Constraints → Format)
- CLAUDE.md를 "에이전트의 헌법"으로 재설계
- Graceful Fallback 패턴 (재시도 → 단순화 → 사람 위임)
- hooks 기반 자동 검증
- **하네스 엔지니어링 심화** (아래 별도 섹션 참조)

**레퍼런스**: Anthropic Prompt Engineering Guide, Claude Code Best Practices

#### 1-A. 하네스 엔지니어링 (Harness Engineering)

> 에이전트의 행동을 코드 바깥에서 제어하는 모든 설정·구조·자동화를 설계하는 영역.
> 프롬프트 엔지니어링이 "무엇을 시킬 것인가"라면, 하네스 엔지니어링은 "어떤 환경에서 실행시킬 것인가"에 해당한다.

##### (1) CLAUDE.md 계층 설계

에이전트에게 주입되는 지시사항을 **스코프별로 분리**하여 관리한다.

| 계층 | 경로 | 역할 | 예시 |
|------|------|------|------|
| 글로벌 | `~/.claude/CLAUDE.md` | 사용자 전역 규칙 | 언어 설정, 응답 스타일, 보안 원칙 |
| 프로젝트 루트 | `./CLAUDE.md` | 프로젝트 공통 규칙 (git 추적) | 코드 컨벤션, 아키텍처 제약, 브랜치 전략 |
| 서브디렉토리 | `./src/CLAUDE.md` | 모듈/레이어별 규칙 | 특정 패키지 의존성 금지, 테스트 패턴 |
| 프로젝트 로컬 | `.claude/settings.local.json` | 개인 환경 (git 미추적) | 퍼미션 설정, 민감한 경로 |

**설계 원칙**:
- 하위 계층이 상위를 **오버라이드**하는 구조 → 범용 규칙은 위로, 예외는 아래로
- CLAUDE.md에는 "왜"를 반드시 포함 — 에이전트가 엣지케이스에서 의도를 추론할 수 있도록
- 정기적으로 `CLAUDE.md audit` 수행: 실제 작업에서 무시된 규칙 제거, 반복 지시사항 승격

##### (2) Hooks 시스템

에이전트의 **도구 호출 전후에 자동 실행**되는 셸 커맨드. 사람이 매번 확인하지 않아도 품질 게이트를 통과시키는 핵심 메커니즘.

| 훅 이벤트 | 트리거 시점 | 활용 패턴 |
|-----------|------------|-----------|
| `PreToolUse` | 도구 실행 직전 | 위험 명령 차단 (force push, rm -rf), 파일 경로 검증 |
| `PostToolUse` | 도구 실행 직후 | lint/format 자동 실행, 테스트 자동 트리거 |
| `Notification` | 에이전트가 사용자 입력 대기 시 | Slack/Discord 알림 연동 |
| `Stop` | 에이전트 응답 완료 시 | 결과 로깅, 메트릭 수집 |

**실전 패턴**:
```jsonc
// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",           // 파일 수정 후
        "command": "npx eslint --fix $CLAUDE_FILE_PATH"  // 자동 린트
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "bash scripts/guard-dangerous-commands.sh"  // 위험 명령 차단
      }
    ]
  }
}
```

**설계 원칙**:
- 훅은 **비파괴적 검증** 우선 — exit code 0이 아니면 도구 실행을 차단
- 훅 실행 시간이 길면 워크플로우 전체가 느려짐 → 무거운 검증은 `Stop` 훅에서 배치로
- 훅 실패 시 에이전트에게 전달되는 메시지를 명확하게 작성 (에이전트가 자가 수정할 수 있도록)

##### (3) 메모리 시스템 설계

에이전트의 **대화 간 지속 기억**을 구조화하여, 매 대화마다 동일한 컨텍스트를 반복 설명하지 않도록 한다.

| 메모리 타입 | 용도 | 갱신 주기 |
|------------|------|-----------|
| `user` | 사용자 역할·선호·역량 | 드물게 (역할 변경 시) |
| `feedback` | 작업 스타일 교정 | 매 대화 (피드백 발생 시) |
| `project` | 진행 중인 이슈·일정·의사결정 | 자주 (상태 변경 시) |
| `reference` | 외부 시스템 포인터 | 드물게 (시스템 변경 시) |

**설계 원칙**:
- `MEMORY.md` 인덱스는 200줄 이하 유지 — 초과 시 오래된 project 메모리부터 정리
- 코드에서 직접 확인 가능한 정보(파일 구조, 컨벤션)는 메모리에 저장하지 않음
- 메모리는 **주장이 아닌 포인터** — "X가 존재한다"가 아니라 "X를 확인해봐라"로 작성

##### (4) 퍼미션 & 신뢰 경계

에이전트에게 허용할 행동의 범위를 명시적으로 설정한다.

| 모드 | 설명 | 적합한 상황 |
|------|------|------------|
| `plan` 모드 | 읽기 전용, 실행 불가 | 아키텍처 리뷰, 탐색 |
| 선택적 허용 | 특정 도구만 자동 승인 | 일상 개발 (Read, Glob, Grep 자동, Bash 수동) |
| `allowedTools` | 화이트리스트 기반 제어 | CI 환경, 자동화 파이프라인 |
| `--dangerously-skip-permissions` | 전체 허용 | 격리된 컨테이너/CI에서만 |

**설계 원칙**:
- 로컬 개발: 읽기·검색 도구는 자동 허용, 쓰기·실행은 확인 후 승인
- CI/자동화: `allowedTools`로 필요한 도구만 화이트리스트, 나머지 전부 차단
- 절대 프로덕션 환경에서 `--dangerously-skip-permissions` 사용 금지

##### (5) MCP 서버 연동

외부 시스템과의 통합을 **Model Context Protocol** 서버로 표준화한다.

| 연동 대상 | MCP 서버 역할 | 활용 |
|-----------|-------------|------|
| PostgreSQL | DB 스키마 조회, 쿼리 실행 | 마이그레이션 검증, 데이터 확인 |
| GitHub | 이슈/PR 조회·생성 | 워크플로우 자동화 |
| Slack | 메시지 전송·채널 조회 | 알림, 리뷰 요청 |
| Linear/Jira | 티켓 상태 조회·갱신 | 작업 추적 연동 |

**설계 원칙**:
- MCP 서버는 **읽기 우선** 설계 — 쓰기 작업은 별도 승인 레이어 추가
- 서버 장애 시 에이전트가 graceful하게 대응하도록 fallback 메시지 설정
- 커스텀 MCP 서버 작성 시 JSON-RPC 표준 준수, 입출력 스키마 문서화

##### (6) 하네스 성숙도 모델

| 레벨 | 상태 | 지표 |
|------|------|------|
| L0 | 기본 설치 | CLAUDE.md 없음, 매번 수동 지시 |
| L1 | 규칙 기반 | CLAUDE.md 작성, 기본 퍼미션 설정 |
| L2 | 자동 검증 | hooks로 린트/테스트 자동화, 메모리 체계화 |
| L3 | 통합 자동화 | MCP 서버 연동, CI 파이프라인 내 에이전트 실행 |
| L4 | 자기 개선 | 에이전트가 CLAUDE.md를 제안·갱신, 훅 성능 모니터링, 메모리 자동 정리 |

**현재 위치 추정**: L1 → L2 전환 중 (hooks 도입 시작, 메모리 구조화 완료)
**단기 목표**: L2 안정화 — 주요 워크플로우에 PreToolUse/PostToolUse 훅 적용 완료

## 3. 포트폴리오 프로젝트

### 프로젝트 0: Claude Code Plugin Marketplace ⭐ (기반 인프라)

- **목적**: agent, skill, command, rule 등 Claude Code 도구를 플러그인 단위로 모듈화·등록·검색·조합하는 개인 마켓플레이스
- **역할**: 모든 워크플로우의 기반 — 여기서 플러그인을 만들고, 각 프로젝트에서 가져다 쓰는 구조
- **핵심**: 플러그인 표준 포맷 정의, 버전 관리, 의존성 해결, 카테고리별 검색
- **어필**: AI 워크플로우를 엔지니어링 수준으로 모듈화·관리하는 설계력

### 프로젝트 1: AI 코드 리뷰 에이전트 파이프라인

- **스택**: Kotlin, Spring Boot, Spring AI, AWS Lambda, Bedrock, GitHub Webhooks, pgvector, Redis
- **핵심**: 대규모 diff 청킹, RAG 기반 팀 맞춤 리뷰, 피드백 루프, 할루시네이션 검증
- **어필**: 실패 관리 설계력 + 엔드투엔드 시스템

### 프로젝트 2: 멀티모달 고객 문의 자동 분류/응답 시스템

- **스택**: Kotlin, Spring Boot, LangChain4j, Bedrock Multimodal, Qdrant, WebFlux, Kafka
- **핵심**: 멀티모달 파이프라인, 신뢰도 기반 에스컬레이션, 비동기 트래픽 대응
- **어필**: AI + 사람 하이브리드 아키텍처

### 프로젝트 3: AI 기반 API 문서 자동 생성/동기화 에이전트

- **스택**: Kotlin, Spring Boot, Spring AI, Claude Code 훅, GitHub Actions, MinIO
- **핵심**: AST + LLM 코드 이해, 부분 업데이트 파이프라인, API 호출 검증
- **어필**: 팀 생산성 향상 도구 설계 (DevEx)
